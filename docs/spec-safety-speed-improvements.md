# Safety & Speed Improvements Specification

## Назначение

Этот документ фиксирует требования к доработкам, которые улучшают безопасность и производительность, не ломая существующую "умную" бизнес-логику FileDO.

Цель: усилить защиту destructive-операций, убрать ложные результаты в поиске дублей, восстановить реальную параллельность хеширования, унифицировать работу с кешем хешей и сократить лишние проходы по дереву файлов при copy.

## Общие принципы

- Не убирать существующие сценарии использования, если они нужны для специальных кейсов проекта.
- Менять поведение по умолчанию только в сторону большей безопасности.
- Не ухудшать работу с поврежденными дисками, медленными носителями и сетевыми путями.
- Все destructive-операции должны иметь предсказуемые guardrails.
- Все изменения должны быть локализованы и сопровождаться простыми manual test сценариями.

## 1. `wipe`: добавить подтверждение

### Проблема

Сейчас `wipe` может уничтожить содержимое целевой директории слишком быстро и без явного пользовательского подтверждения. Для корневых и пограничных путей риск особенно высокий.

### Требования

- Перед запуском `wipe` всегда показывать предупреждение с точным путем.
- Требовать явный ввод подтверждения от пользователя.
- Для опасных путей требовать усиленное подтверждение.

### Опасные пути

- Корень локального диска: `C:\`, `D:\`
- UNC root и верхний уровень network share
- Junction, symlink, reparse point
- Папка, совпадающая с TEMP/FileDO working dir, если это может затронуть служебные файлы

### Предлагаемое поведение

- Обычный `wipe`:
  - показать путь
  - показать количество найденных объектов, если это можно получить быстро
  - спросить `Type WIPE to continue:`
- Для опасных путей:
  - либо запрещать по умолчанию
  - либо требовать второе подтверждение с точным путем
- Для automation:
  - разрешить явный флаг вида `--yes` или `--force-wipe`
  - этот флаг должен обходить только интерактивный prompt, но не базовые проверки на явно опасные цели

### Non-goals

- Не менять сам алгоритм fast wipe vs standard wipe, кроме добавления guardrails.

### Acceptance criteria

- Без подтверждения `wipe` не стартует.
- Для root/share root/reparse point есть отдельная защита.
- Поведение документировано в help/README.

## 2. Исправить поиск дублей

### Проблема

Кеш хешей сейчас может повторно использовать хеш для файла, который изменился, но сохранил тот же размер. Это может привести к ложному совпадению дублей и опасным delete/move действиям.

### Требования

- Кеш валиден только если совпадают:
  - путь
  - размер
  - время модификации
- Желательно дополнительно учитывать Windows file identity, если это можно сделать без сильного усложнения.
- Перед destructive-обработкой дублей желательно повторно убедиться, что файл не изменился после сканирования.

### Минимальный безопасный вариант

- Обновить `CacheEntry`, чтобы он хранил `ModTime`.
- При чтении из кеша считать entry валидным только при совпадении `Size` и `ModTime`.
- При сохранении нового хеша всегда обновлять `LastSeen` и `ModTime`.

### Acceptance criteria

- Изменение содержимого файла при сохранении прежнего размера больше не использует старый cached hash.
- Duplicate delete/move работает только по актуальным данным.

## 3. Исправить параллельность при поиске дублей

### Проблема

Сейчас логика `GetHash()` сама вычисляет хеш на cache miss, из-за чего значительная часть работы идет синхронно и worker pool теряет смысл.

### Требования

- Разделить операции:
  - `LookupHash(...)` только читает кеш
  - `ComputeHash(...)` выполняется worker'ами
  - `StoreHash(...)` записывает результат в кеш
- В `FindDuplicates()` поток должен быть таким:
  - scan files
  - try cache lookup
  - misses -> queue to workers
  - results -> aggregate + store in cache

### Ожидаемый эффект

- Реальная загрузка нескольких worker'ов на quick hash и full hash стадиях
- Снижение wall-clock time на больших деревьях
- Более предсказуемая progress-индикация

### Acceptance criteria

- На cache miss основной goroutine не считает хеш сама.
- Все miss-файлы действительно проходят через worker pool.
- Нет потери результатов и race condition при закрытии каналов.

## 4. Исправить inconsistent path logic у кеша хешей

### Проблема

Сейчас загрузка и сохранение кеша используют разные базовые пути, из-за чего кеш может "теряться" между запусками.

### Требования

- Вынести единый resolver, например `GetHashCachePath()`.
- И чтение, и запись должны использовать один и тот же путь.
- Путь должен быть предсказуем:
  - либо рядом с exe
  - либо в current working directory
  - либо в выделенной app-data директории

### Рекомендация

Для CLI-инструмента безопаснее и предсказуемее использовать путь рядом с exe или в специальной app-data папке, но не смешивать обе модели.

### Дополнительно

- Запись делать атомарно: `hash_cache.json.tmp` -> rename
- Не повреждать кеш при прерывании процесса

### Acceptance criteria

- После успешного запуска кеш читается и пишется в одном и том же месте.
- Перезапуск приложения использует тот же кеш.

## 5. Убрать лишние проходы по дереву в `copy`

### Проблема

Сейчас copy-режимы делают лишние полные `Walk` по дереву: отдельно для подсчета, отдельно для сбора файлов, отдельно для копирования. На больших каталогах это заметно замедляет старт.

### Требования

- Сократить количество полных проходов по дереву там, где это не ломает progress и special logic.
- Разделить режимы:
  - `accurate progress` — допускает precount
  - `fast start` — начинает копирование сразу

### Предлагаемый минимальный вариант

- Для обычного `copy`:
  - объединить scan и dispatch, если не нужен строгий total upfront
- Для `fastcopy`:
  - один проход собирает `FileJob` и одновременно считает totals
  - директории создаются по мере обхода
- Если нужен точный ETA:
  - precount оставлять только по явному флагу или для verbose mode

### Ограничения

- Не ломать current damaged-disk handling
- Не ломать skip damaged files logic
- Не ломать small file batching и large file scheduling

### Acceptance criteria

- Для стандартного копирования число полных `Walk` уменьшено.
- Начало реального копирования происходит раньше.
- На больших деревьях сокращается время "до первого полезного действия".

## Порядок внедрения

1. `wipe` confirmation и guardrails
2. Единый cache path
3. Разделение cache lookup / worker hashing
4. Валидация cached hash по `size + modtime`
5. Оптимизация tree walk в copy

## Проверка после реализации

- Manual test: `wipe` на обычной папке, root path, network path
- Manual test: duplicate scan с измененным файлом того же размера
- Manual test: duplicate scan на большом наборе файлов с проверкой реальной загрузки worker'ов
- Manual test: повторный запуск duplicate scan с reuse кеша
- Manual test: copy большого дерева с замером времени до старта реального копирования
